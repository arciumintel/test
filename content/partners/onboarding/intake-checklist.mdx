Arcademy staff uses your inputs to build learner-facing content. **Complete, accurate inputs speed up publishing.**

## Project basics

- **Official name** — Exactly as it should appear on Arcademy
- **Short description** — One or two sentences; avoid assuming crypto or privacy expertise
- **Longer description** — What the project does, who it helps, why it matters in Arcium
- **Category** — e.g. application, protocol, integration, infrastructure, tooling, service
- **Primary user segment** — Who the course serves first (new Solana user, privacy-curious app user, etc.)
- **Primary problem solved** — Plain-language user problem
- **Learning outcomes** — 3–5 ideas learners should understand after the course

## Links and assets

| Asset | Notes |
| --- | --- |
| Website, docs, app entry | Current URLs only |
| GitHub, social, community | Optional but helpful |
| Logo and screenshots | Must be **approved for Arcademy use** |
| Demo video | Optional |
| Brand guidelines | If you have restrictions on visual treatment |

Arcademy hosts media through Cloudinary in V1; you provide approved assets and links.

## Referral goals

- **Desired learner action after completion** — e.g. try the app, join Discord, read a specific guide
- **Preferred referral URL** — Main "next step" link from the project page
- **Where you plan to link Arcademy** — Docs, onboarding, Discord, campaigns, help center
- **UTM preferences** — Optional; staff can define a structure if you do not have one

## Course scope

- **Course type** — Foundational, project onboarding, builder intro, or feature-specific
- **Working title and summary** — Learner outcome in plain language
- **Estimated length** — Typically **20–45 minutes** in V1 unless there is a clear reason to go longer
- **Prerequisites** — Informational only in V1 (not hard gates)
- **Draft lesson sequence** — Ordered list of 3–5 lessons with what each should teach

Example lesson outline:

| Order | Lesson | Learner takeaway |
| --- | --- | --- |
| 1 | What is [Project]? | Purpose and who it serves |
| 2 | Why it matters | Problem and value in Arcium |
| 3 | First steps | How to get started safely |
| 4 | Common pitfalls | Mistakes to avoid |
| 5 | What's next | Path after the course |

## Source material

Provide staff with:

- Required reading (overview, onboarding, FAQ, support themes)
- A **plain-language explainer** (5–8 sentences; does not need to be polished)
- A glossary of terms that need careful explanation
- **Claims requiring accuracy review** — privacy guarantees, security assumptions, launch status, performance numbers

## Beginner signals

Help staff prioritize what the course must address:

- Top beginner questions
- Common mistakes or wrong expectations
- Examples from Discord, support, or community feedback

## Quiz inputs

V1 courses use one **required final quiz** (single-select multiple choice, unlimited retries, instructional explanations):

- 5–8 learning objectives to assess
- Suggested questions in plain language (no trick questions)
- Concepts to **avoid** testing (too advanced, unstable, or out of scope)
- Passing threshold — default **70%** unless you request otherwise with justification

## Badge inputs

- Badge name and one-sentence description
- Criteria language (complete lessons + pass quiz + connect wallet)
- Image direction and attribution wording
